# 02. アーキテクチャ

## 技術スタック

| 領域 | 選定 | 理由 |
|---|---|---|
| 基盤 | Vite + React 18+ + TypeScript (strict) | エディタUIの量産性、エコシステム |
| 状態管理 | Zustand + immer | 軽量。ドキュメント変更はコマンド経由のみ |
| ステージ描画 | **PixiJS v8** (WebGL) | フラットなベクタ形状をGraphicsで直接描画。書き出しと同一レンダラ |
| パートエディタ描画 | **SVG (React)** | 頂点編集・ピン打ちのUXはDOMイベントが圧倒的に楽 |
| 動画書き出し | **WebCodecs** (VideoEncoder/AudioEncoder) + mp4-muxer | ffmpeg不要でH.264+AACのMP4が出る |
| 音声 | Web Audio API | デコード・波形・RMS解析(リップフラップ用) |
| ファイル | File System Access API | プロジェクトフォルダを掴んでローカル完結 |
| テスト | vitest + Playwright(ゴールデンフレーム比較: pixelmatch) | 後述のテスト戦略参照 |
| 対象環境 | Chrome / Edge 最新(macOS) | WebCodecs・FS Access前提。Safari非対応と明記 |

**配布形態**: 当面はローカルWebアプリ(`pnpm dev`)。FS Accessの不便やエンコード安定性が問題になったらElectron/Tauri化を検討(判断はPhase 5以降)。

## レイヤー構成

```
┌─────────────────────────────────────────┐
│ editor-ui     Reactパネル / SVGパートエディタ / タイムライン │
├─────────────────────────────────────────┤
│ render-pixi   FrameState → Pixi表示(プレビュー/書き出し共用)  │
├─────────────────────────────────────────┤
│ runtime       評価器: (doc, sceneId, t) → FrameState        │
│               ポーズ補間・表情解決・髪物理・カメラ            │
├─────────────────────────────────────────┤
│ core-model    スキーマ型・コマンド・undo・バリデーション       │
├─────────────────────────────────────────┤
│ io            保存/読込・アセット解決・エンコード             │
└─────────────────────────────────────────┘
```

- `core-model` と `runtime` は **DOM非依存の純TypeScript**。headlessでテスト可能
- **評価器が唯一の真実**: プレビュー・書き出し・コンタクトシートはすべて同じ `evaluate()` を通る。「プレビューと書き出しで見た目が違う」を構造的に排除する

## 決定論(最重要原則)

書き出しの再現性と回帰テストのため、runtime層では:

- `Date.now()` / `Math.random()` 禁止。乱数は**シーン毎の固定seed**(まばたき間隔等)
- 髪物理は**固定タイムステップ dt=1/120** のサブステップ。常にシーン先頭(t=0)からシミュレートした状態を評価
- フレーム評価は `evaluate(doc, sceneId, t)` の純関数(物理はシーン単位でキャッシュ可)

## 座標系・単位

- y-down、ステージ論理解像度 **1920×1080**
- キャラは標準身長 **600 units** で作画(テンプレートが基準を提供)。ステージ配置時にscale
- パーツ形状とピンは**キャラ空間**(全身キャンバスの座標、root=原点)で保持。描画時に `W_bone ∘ T(−originピン)` でボーン追従に変換(ゴースト下絵の上にそのまま描け、ミラー複製が単純なx反転になるための判断)

## Undo/Redo

- コマンドパターン: `Command { do, undo, merge? }`
- **ドキュメント毎に独立スタック**(キャラエディタとシーンエディタは別)
- 連続ドラッグ・カラー変更は `merge` で1コマンドに統合
- 全ドキュメント変更はコマンド経由のみ(直接mutationはlintで禁止)

## レンダリング戦略

- パーツのパスデータ → Pixi Graphics で塗り描画
- キャラはコンテナ単位で `cacheAsTexture`、表情/ハンドシェイプ差し替え・ズーム閾値超過時のみ無効化
- ミラー反転はキャラルートの scaleX=-1(テキスト等の非反転フラグは将来)

## テスト戦略(モデル戦略 09 と連動)

1. **vitest**: 補間・トランスフォーム・物理・コマンドの数値テスト
2. **ゴールデンフレーム回帰**: フィクスチャキャラ×代表ポーズをheadless Chromeで描画 → pixelmatchで自動比較。**greenの間はAIの目視不要**
3. **コンタクトシートharness**: `キャラ × ポーズ4種 × 表情6種` を1枚のグリッドPNG(1280px幅)に出力するCLI。新機能・差分発生時にfable5が**1コールで全体を視覚検証**するための装置。ユーザー向けプレビュー機能としても搭載

## オートセーブ

60秒毎+操作アイドル時に `project.autosave.json` へ。クラッシュ復帰ダイアログ。

## パフォーマンス予算

- プレビュー: キャラ5体+小道具20でも60fps(M系Mac)
- 書き出し: 1080p30が実時間の2倍以内
- キャラファイル < 500KB、起動 < 2秒
