# 09. モデル戦略(fable5 / opus / sonnet の割り当て)

## 基本原則

> **fable5は「目」と「背骨」。opusは「重い実装の腕」。sonnetは「量産の手」。**

| モデル | 担当領域 | 理由 |
|---|---|---|
| **fable5** (claude-fable-5) | ①仕様・アーキテクチャの意思決定 ②支柱の難所実装: ピン×スケルトンの座標変換、表情合成、髪物理、クリップ評価器 ③**視覚検証**(コンタクトシート・フレーム画像のレビュー) ④原因不明のクロスカッティングなバグ(座標系・z順・書き出し色ズレ等) | 画像認識力と複雑な空間推論が必要な箇所。ここをケチると手戻りで逆に高くつく |
| **opus** (claude-opus-4-8) | 大型だが定石のある実装: タイムラインUI、undo統合、シーンエディタの状態管理、WebCodecs書き出しパイプライン | 複雑だが視覚判断より構造設計が支配的な領域 |
| **sonnet** (claude-sonnet-4-6) | UIコンポーネント量産、パネル/フォーム、CRUD、JSONスキーマとシリアライズ、プリセットクリップのデータ打ち込み、テスト作成、リファクタ、アセットSVG量産、ドキュメント | 仕様が明確なら品質十分。物量はここで稼ぐ |

トークン配分の目安: **fable5 : opus : sonnet ≈ 25 : 35 : 40**(フェーズにより変動。Phase 1-2はfable5比率が上がり、Phase 4以降は下がる)

## fable5のトークンを節約する4つの仕組み

「画像認識はfable5に任せたいがフルには使えない」への構造的な解:

### 1. コンタクトシート方式(視覚検証の圧縮)

スクリーンショットを1枚ずつ見せる代わりに、**`ポーズ4種 × 表情6種` 等を1枚のグリッドPNG(幅1280px)に合成して1コールでレビュー**する。tools/contactsheet CLIをPhase 2で整備(02参照)。
- 24状態の確認が画像1枚=1回のfable5呼び出しで済む
- NG項目だけ個別拡大して再レビュー(2段階方式)

### 2. ピクセル回帰ゲート(目視の発動条件を絞る)

ゴールデンフレーム回帰(pixelmatch)が**greenの間はAIの目視自体が不要**。fable5が画像を見るのは:
- 新機能の初回品質判定
- 回帰テストが差分を検出した時
- アートディレクション判断(アセット検収)
の3場面のみ。日常の開発ループでは画像を一切流さない。

### 3. 仕様書のモジュール分割(コンテキスト最小化)

本仕様書はフェーズ毎に必要なファイルだけ読む前提で分割済み(08の対応表参照)。セッション開始時に全部読まない。

### 4. 設計と実装の分離(モデル間ハンドオフ)

```
fable5: インターフェース・型・受入条件・落とし穴メモを書く(設計)
   ↓
sonnet/opus: モジュール実装(Agentで並列も可)
   ↓
自動テスト(vitest + ゴールデン回帰)
   ↓ green → 終了(fable5は登場しない)
   ↓ 差分 or 新機能
fable5: コンタクトシート1枚で視覚判定 → NG箇所の修正指示のみ返す
```

## フェーズ別の運用イメージ

| Phase | セッションの回し方 |
|---|---|
| 0 | sonnet単独。最後にfable5がアーキレビュー1回 |
| 1-2 | **fable5主導**。座標変換・物理・表情合成は自ら実装。UI雑務はsonnetにAgent委譲。節目でコンタクトシート自己検証 |
| 3 | fable5が評価器とwalk/runを作り込み → クリップ残り8本はsonnetがデータ量産 → コンタクトシートで一括検収 |
| 4-5 | opus主導。fable5はUI/書き出しの節目スクショ確認のみ(各フェーズ2〜3回) |
| 6 | sonnet主導でアセット量産 → fable5がシート検収。Replace等の機能はopus |

## 画像をfable5に渡す時の規約

- 幅1280px以下に縮小(細部確認時のみ等倍切り抜き)
- 1回の確認でなるべくグリッド合成(states × variants)
- 動きの確認は動画ではなく**キーフレーム抽出(0%, 25%, 50%, 75%)の横並び画像**で代替。それで不足する「揺れの気持ちよさ」だけGIF/連続フレームを使う

## アプリ内AI機能(Phase 7・ランタイムのAPI利用)

開発時とは別軸で、ツール自体がClaude APIを呼ぶ将来機能。低頻度・高価値に絞る:

| 機能 | モデル | 頻度 | 設計メモ |
|---|---|---|---|
| 参考画像→パレット抽出+パーツ構成提案 | claude-fable-5 | キャラ作成時に1回 | 画像1枚→6スロットの色+「ボブで前髪あり」等の構造提案JSON。fable5の画像認識を活かす本命 |
| 台本→シーン分割・絵コンテ下書き | claude-sonnet-4-6 | 動画毎に1回 | テキスト→scenes[]のドラフト生成(キャラ配置・表情・アクション提案) |
| セリフ→表情/アクション自動提案 | claude-haiku-4-5 | セリフ毎 | 「えっ!?」→surprised+jump 等の軽量マッピング |
| アセットタグ自動付与 | claude-haiku-4-5 | 取り込み時 | 検索性向上 |

- すべて「提案→ユーザー確認」のドラフト方式(自動確定しない)
- APIキーはローカル設定。コスト表示を必ずUIに出す
